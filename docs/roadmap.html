구현 로드맵 — Peekaboo

스캐폴딩부터 배포까지 5개 페이즈로 나눈다. 각 페이즈는 산출물·검증 게이트(실제 위협 시나리오 통과)·리스크를 갖는다. 핵심 원칙은 "띄우고 즉시 숨긴다"를 가장 먼저(MVP) 동작시키고, 스텔스를 단계적으로 쌓는 것이다. 기능 정의는 기능 명세, 동작 근거는 아키텍처 참조.

①페이즈 타임라인

각 페이즈는 이전 페이즈의 검증 게이트를 통과해야 다음으로 넘어간다. 전역 정책(단계별 최대 5파일, 승인 후 진행)에 맞춰 페이즈를 잘게 쪼갰다.

Phase 0 · 스캐폴딩

지금

산출물 — npm create tauri-app로 프로젝트 생성, cargo tauri add global-shortcut, macos-private-api feature 활성, capabilities/default.json에 필요한 권한 선언. 빈 투명 창이 떠서 tauri dev가 도는 상태.

검증 게이트 — 투명·무테 창이 화면에 뜨고(배경 비침), 콘솔 에러 없음.

리스크 투명이 안 보이면 macOSPrivateApi+feature 누락 의심.

Phase 1 · MVP — 오버레이 + 패닉

지금

산출물 — 웹툰 URL 로드, 항상-위/크기·위치, 그리고 패닉 글로벌 단축키(토글 hide/show). 투명도 기본값 적용. "보고 → 즉시 숨긴다" 수직 경로 완성.

검증 게이트 — 다른 앱이 포커스인 상태에서 패닉 키를 누르면 창이 즉시 사라지고, 다시 누르면 보던 위치로 복귀.

리스크 플러그인 이중 초기화(os error 22) · 미디어키 선택 금지.

Phase 2 · 스텔스 본체 — Hover-Reveal

다음

산출물 — 클릭 통과(set_ignore_cursor_events) + 커서 폴링 루프(cursor_position)로 핫존 hover-reveal, 평소/호버 투명도 분리, 노출/패닉 상태 머신(Rust 단일 소스) 구현.

검증 게이트 — 핫존 밖 클릭이 아래 앱으로 통과되고, 핫존에 커서를 올리면 또렷해지며, 폴링 CPU 사용이 수용 범위.

리스크 폴링 비용 · 핫존 rect 좌표(스케일/멀티모니터) 계산 정확도.

Phase 3 · 은폐 강화

다음

산출물 — Accessory(Dock·Cmd-Tab 숨김) + set_focus() 온디맨드 포커스, Spaces 전역(visibleOnAllWorkspaces), content protection 적용 및 한계 실측(macOS 버전별 화면공유 노출 확인).

검증 게이트 — Dock/Cmd-Tab에 안 뜨고, 입력 필요 시 포커스가 잡힘. content protection이 현 macOS에서 화면공유를 막는지/못 막는지 실측 기록.

리스크 content protection macOS 15+ 무효(확정) · Accessory 포커스 마찰 · 풀스크린 위 부유 한계.

Phase 4 · 폴리시 + 배포

이후

산출물 — 설정 UI(크기·투명도·핫존·단축키·북마크), 상태 영속화(store/fs), 마지막 위치 복원, macOS 빌드·개인용 미서명 실행 경로 정리(필요 시 서명·공증).

검증 게이트 — 재실행 시 설정·북마크·마지막 위치 복원, 빌드 산출물이 본인 머신에서 정상 실행.

리스크 서명/공증 환경변수 변동(확인필요) · 영속화 파일이 흔적이 되지 않도록.

②검증 게이트 (위협 시나리오)

스텔스 앱은 "기능이 된다"가 아니라 "들키지 않는다"가 합격선이다. 각 게이트는 실제 위협 상황을 흉내 내어 통과 여부를 판정한다.

게이트시나리오판정 기준페이즈
즉시 탈출다른 앱에서 작업 중 패닉 키 입력창이 체감 지연 없이 사라짐P1
복귀 무결성패닉 후 다시 표시보던 위치·스크롤 유지P1
클릭 통과핫존 밖에서 아래 앱 클릭클릭이 아래로 통과P2
호버 노출핫존에 커서 진입/이탈또렷↔고스트 전환P2
앱 은폐Dock·Cmd-Tab·메뉴바 확인어디에도 안 보임P3
화면공유Zoom/Meet/QuickTime로 화면 공유△ 버전 의존 — macOS 15+는 노출됨(기록)P3
화면공유 게이트는 "통과"가 아니라 "기록"이다

content protection이 macOS 15+에서 동작하지 않음은 확정 사실(근거)이므로, 이 게이트의 목적은 "막는다"가 아니라 "어느 환경에서 노출되는지 정확히 알고 사용자에게 알린다"이다. 실질 방어는 P1의 패닉 키(화면 공유 전/중 사전 숨김)로 한다.

③리스크 레지스터

영향·가능성과 완화책. 1번이 가장 중대하며 제품 메시징에 직접 영향을 준다.

#리스크영향가능성완화책
1 content protection이 macOS 15+에서 화면공유를 못 막음 높음 확정 패닉 키 사전 숨김 워크플로를 1차 방어로 · 사용자에게 한계 명시 · (v2) 화면공유 시작 감지 시 자동 패닉 가능성 조사
2 Accessory 앱의 포커스 마찰(show()만으론 입력 안 됨) 중간 중 입력 필요 시 set_focus() 명시 · 거슬리면 NSPanel 검토(아래 5)
3 웹툰 사이트의 로그인/임베드 차단 중간 중 메인 윈도우 내비게이션으로 로드 · 자주 보는 사이트별 사전 검증 · 안 되면 대체 뷰어
4 커서 폴링 CPU 부담 낮음 중 40ms 주기·엣지 트리거 · 정지 시 적응형 주기 · 핫존만 통과해제로 폴링 축소
5 풀스크린 위 부유·비활성 부유엔 NSPanel 필요(스톡 미노출) 중간 낮음~중 우선 스톡 창 + set_focus로 진행 · 실측 불편 시 objc2/커뮤니티 플러그인 도입(비용 인지)
6 투명=macos-private-api → App Store 배포 불가 낮음 확정 개인 도구라 영향 없음 — 직접 배포/미서명 경로
7 구현 함정(이중 초기화·미디어키·import 누락) 낮음 낮음 단일 등록 · 일반 단축키 조합 · Emitter 등 트레이트 import 체크

④미결정 사항

진행 중 결정이 필요한 분기점. 대부분 "실측 후 결정"으로 미뤄도 되는 항목이다.

NSPanel 도입 여부

P3 결정

스톡 창 + set_focus()의 "활성 전환"이 실사용에서 거슬리는지 실측한 뒤, 거슬리면 NSPanel(nonactivatingPanel) 도입. objc2/커뮤니티 플러그인 비용 발생.

macOS 15+ 화면공유 대응

조사

content protection이 무효인 환경에서, "화면 공유 시작 감지 → 자동 패닉"이 신뢰성 있게 가능한지 조사(v2). 불가하면 사용자 안내·수동 패닉 원칙 유지.

프론트 프레임워크

P0 결정

표현이 단순하므로 vanilla TS로 충분할 수 있음 vs 설정 UI 편의를 위해 경량 프레임워크. 스캐폴드 시 확정.

영속화 수단

P4 결정

tauri-plugin-store(간단 KV) vs fs 직접. 흔적 최소화(파일 위치/이름) 관점도 함께 고려.

한 줄 요약

P1에서 "띄우고 즉시 숨긴다"를 먼저 완성(가장 쉽고 가장 가치 큼) → P2에서 클릭통과+hover-reveal로 평소 은닉 → P3에서 앱 은폐와 화면공유 한계 실측 → P4에서 설정·배포. 화면 공유 비노출은 보장하지 못한다는 사실을 처음부터 설계·메시징에 반영하고, 그 자리를 패닉 키가 메운다.
