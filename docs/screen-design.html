화면 설계 — Peekaboo

Peekaboo는 제목표시줄·메뉴·툴바 같은 일반 앱 chrome이 없다. 따라서 "화면"은 곧 오버레이의 상태(고스트·노출·패닉)와 설정 패널로 정의된다. 이 문서는 그 화면들을 손그림이 아니라 실제 렌더되는 HTML/CSS·SVG 목업으로 구현해, 보이는 모습과 인터랙션을 확정한다. 기능 정의는 기능 명세, 상태 전이의 정식 정의는 아키텍처 상태 머신 참조.

고스트 = 평소 노출 = 호버 패닉 = 소멸 chrome 없음 · 미니 툴바

①화면 구성 원칙

스텔스 앱이므로 "잘 보이는 UI"가 아니라 "안 띄는 UI"가 목표다. 화면 설계의 네 원칙이 모든 목업을 지배한다.

Chrome 없음

원칙

제목표시줄·테두리·그림자·작업표시줄 항목을 모두 제거. 콘텐츠만 화면에 얹힌 듯 보인다. 조작 요소는 호버 시에만 드러나는 미니 툴바로 최소화.

평소엔 유령, 호버엔 또렷

원칙

기본 상태는 낮은 불투명도 + 클릭 통과(고스트). 핫존에 커서가 들어올 때만 또렷해지고 상호작용을 받는다.

위기엔 즉시 소멸

스텔스 핵심

패닉 단축키 한 번으로 화면에서 완전히 사라진다. 잔상·흔적 없이. 어떤 화면 상태에서든 동일하게 동작.

주변과 덜 튀게

원칙

다크·저채도 톤으로 업무 화면 위에서 시선을 덜 끈다. Dock·Cmd-Tab·메뉴바 흔적은 숨김(Accessory).

②데스크톱 맥락 · 3상태

업무 화면(예: 스프레드시트) 위에 오버레이가 얹힌 실제 맥락. 같은 장면을 평소(고스트) → 호버(노출) → 패닉(소멸) 세 상태로 보여준다. 이 세 컷이 제품 전체 경험을 요약한다.

③오버레이 창 해부

오버레이 한 장의 구성 요소. 무테 창이라 드래그·리사이즈·조작을 위한 영역을 직접 설계해야 한다. 미니 툴바는 호버(노출) 상태에서만 나타난다.

④설정 패널

미니 툴바의 ☰(또는 설정 단축키)로 호출되는 패널. 크기·투명도·핫존·패닉 단축키·은폐 토글·북마크를 한곳에서 조정한다. 이 패널 자체도 호출 시에만 뜨고 닫으면 사라진다.

컨트롤 설계 노트

  • 투명도 2종 — 평소(낮게)·호버(높게)를 분리해 "안 보임 ↔ 또렷"의 폭을 사용자가 정한다.
  • 핫존 위치 — 3×3로 간단히 지정(추후 자유 좌표). 드래그 핸들과 동일 영역.
  • 패닉 단축키 — 키 캡처 필드. 일반 modifier+key만 허용(미디어키 입력 시 거부 — 권한 이슈).
  • 화면공유 비노출 토글 — 켜되 best-effort 태그로 한계를 명시(macOS 15+ 무효, 근거).
설정을 어떻게 여는가

Dock·메뉴바를 숨기므로(Accessory) 설정 진입은 ⑴ 오버레이 호버 → 미니 툴바 ☰, ⑵ 전용 단축키(예: ⌘⇧,) 두 경로로 둔다. 설정 창은 Accessory 특성상 입력을 받으려면 set_focus()가 필요하다(포커스 모델).

⑤투명도 스펙트럼

같은 콘텐츠가 불투명도에 따라 어떻게 읽히는지. "평소"는 보통 8–20%(존재만 인지), "호버"는 90–100%(편하게 읽기)로 설정한다. 그 사이를 사용자가 취향대로 조절한다.

왼쪽일수록 "있는 줄도 모를 정도", 오른쪽일수록 "편하게 읽기". 평소값을 낮출수록 안전하지만 흘끗 볼 때 불편하므로, 사용자가 자기 환경(자리 배치·모니터 각도)에 맞춰 균형점을 잡게 한다. 이 전환은 호버 진입/이탈에 .12s 정도로 부드럽게 보간한다.

⑥컨트롤 접근 · 화면 매핑

각 화면이 어떤 기능·동작 문서에 대응하는지 정리한다. 화면 설계는 기능·아키텍처의 시각적 구현이다.

목업의 위치

위 목업은 레이아웃·상태·인터랙션을 확정하기 위한 것으로, 실제 구현 시 그대로 옮길 수 있는 형태(투명창 위 CSS opacity, 호버 시 툴바 표시, 핫존 점선)다. 색·간격은 다크·저채도 원칙을 따르되 사용자 환경에서 튀지 않게 추후 미세조정한다.
