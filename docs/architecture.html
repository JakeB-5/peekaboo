아키텍처 · 동작 설계 — Peekaboo

Peekaboo의 난이도는 UI가 아니라 macOS 창 동작 제어에 있다. 이 문서는 Tauri 2-프로세스 모델에서 스텔스의 본체(Rust 코어)와 표현(WebView)의 책임 경계를 확정하고, 노출/패닉 상태 머신, 창 속성 제어, 그리고 가장 까다로운 클릭통과 하 hover 감지 설계를 검증된 API로 정의한다. 기능 단위 요구는 기능 명세, 정확한 코드·설정은 기술스택·구조 참조.

Rust 코어 = 스텔스 본체 WebView = 표현 상태 단일 소스 = Rust content protection 한계

①프로세스 모델 · 책임 경계

Tauri 앱은 Rust로 작성한 네이티브 코어 프로세스와, 시스템 WebView에서 도는 프론트엔드로 나뉜다. Peekaboo에서 스텔스를 결정하는 모든 창 속성 제어는 Rust 코어가 소유하고, WebView는 콘텐츠 표현과 상호작용만 담당한다. 이 경계가 흐려지면(예: 패닉을 JS에서 처리) 지연·신뢰성이 나빠진다.

②노출 · 패닉 상태 머신

Peekaboo의 동작은 네 상태와 그 사이 전이로 환원된다. 이 상태 머신은 Rust 코어가 단일 소스로 보유하고, 전이가 일어날 때마다 프론트로 이벤트를 emit해 투명도/표현을 맞춘다.

상태별 창 속성

Hidden — window.hide()(화면에서 제거). Ghost — 낮은 CSS 불투명도 + set_ignore_cursor_events(true)(클릭 통과). Revealed — 높은 불투명도 + 핫존에서 set_ignore_cursor_events(false). Focused — Accessory 앱이므로 입력 수신 전 set_focus()로 온디맨드 활성화. 각 전이는 Rust가 네이티브 API를 적용하고 프론트에는 표현 동기화 이벤트만 보낸다.

현실 점검 · content protection (근거)

검증 근거: Tauri는 set_content_protected(true)를 macOS NSWindow.sharingType = NSWindowSharingNone으로 매핑한다. 그러나 macOS 15(Sequoia)+에서는 ScreenCaptureKit가 모든 NSWindow를 단일 합성 프레임버퍼로 캡처하며 이 플래그를 무시한다(Tauri #14200, "status: upstream", 알려진 우회 없음). 결과적으로 Zoom·Meet·QuickTime 화면 공유에 창이 노출된다. content protection은 macOS ≤14 + 구형 CoreGraphics 캡처에서만 유효하고, 일부 Chromium 기반 공유는 구버전에서도 못 막는다(#10213). 설계 결론: 화면 공유 비노출은 보장 기능이 아닌 보조 수단으로 두고, 실질 방어는 패닉 단축키로 한다. 대응 전략은 로드맵 리스크 참조.

④Hover 감지 설계 (클릭통과 크럭스)

"클릭은 아래로 통과시키되 호버는 감지"는 본질적으로 충돌한다. set_ignore_cursor_events(true)를 켜는 순간 webview는 마우스 이벤트를 전혀 받지 못하므로 DOM mouseenter/mouseleave는 물론 네이티브 WindowEvent::CursorEntered/Left도 발생하지 않는다. 검증된 유일한 정공법은 전역 커서 좌표 폴링 + 핫존 사각형 히트테스트다.

설계 결정

  • 위치 = Rust 루프 — cursor_position()(desktop-only)를 코어 스레드에서 폴링. JS setInterval + cursorPosition()도 가능하나 매 틱 IPC라 비효율.
  • 주기 — 30–60ms(타협: 반응성 vs CPU). 정지 시 주기를 늘리는 적응형도 가능.
  • 핫존 rect — outer_position() + inner_size()(+ scale_factor)로 창 좌표를 구해 핫존 사각형을 계산.
  • 엣지 트리거 — "내부↔외부"가 바뀌는 순간만 네이티브 토글·이벤트 emit.

대안 · 한계

  • 핫존만 통과해제 — 작은 핫존 영역은 클릭통과를 끄고, 나머지만 통과 → 폴링 부담 경감(레이아웃 설계 필요).
  • 이벤트 기반 — objc2로 NSEvent 전역 모니터를 달면 폴링 없이 가능하나, 손쉬운 사용 권한이 필요해 스텔스/마찰 측면에서 비권장.
  • 공식 한계 — "클릭통과 창과 커서 교차" 전용 이벤트는 stable Tauri에 없음(feature #9250가 cursor_position 폴링으로 귀결).

⑤명령 · 이벤트 계약

프론트(표현)와 코어(상태) 사이 인터페이스. 명령은 JS→Rust(invoke), 이벤트는 Rust→JS(emit). 상태 머신은 Rust 단일 소스이므로, 프론트는 명령을 "요청"하고 결과는 이벤트로 받는다.

명령 (JS → Rust)

  • set_opacity(ghost, revealed) — 평소/호버 불투명도 설정
  • set_hotzone(rect) — 핫존 위치·크기 갱신
  • set_panic_shortcut(accel) — 패닉 단축키 재등록
  • toggle_panic() — 수동 토글(트레이/버튼)
  • set_content_protected(bool) — 보조(한계 인지)
  • load_url(url) · save_bookmark(...)

이벤트 (Rust → JS)

  • reveal-state-changed → { state: "ghost"|"revealed"|"hidden" } — 표현(불투명도) 동기화. Focused는 입력 포커스 전용 전이라 불투명도 변화가 없어 이 이벤트에는 포함하지 않는다.
  • panic — 숨김 직전 잔상 제거 트리거
  • shortcut-registered → { ok, accel }
  • settings-loaded → 영속화 상태 주입
패닉의 지연 최소화

패닉은 "이벤트 → 프론트 처리 → 명령 → 숨김" 왕복을 타면 느리다. 따라서 글로벌 단축키 핸들러(Rust)에서 곧장 window.hide()를 호출하고, 잔상 제거가 필요하면 panic 이벤트는 숨김과 병행해 보낸다. 프론트가 죽어 있어도 숨김은 동작한다.

⑥포커스 모델 (Accessory)

앱을 Dock·Cmd-Tab에서 숨기면(Accessory) 입력 포커스 거동이 달라진다. 검증된 사실에 근거해 "평소엔 비활성, 필요할 때만 활성"을 설계한다.

확정 사실

  • Accessory 앱도 포커스 수신 가능 — Apple 정의상 "프로그램적으로 또는 창 클릭으로 활성화될 수 있음".
  • 글로벌 단축키(패닉)는 activation policy와 무관하게 발화 — Accessory여도, 비포커스여도 동작.
  • 입력이 필요하면 set_focus()로 앱을 프로그램적으로 활성화(내부적으로 앱 활성화 + 키 윈도우 지정에 해당)해 키보드 입력을 받음.

마찰 · 주의

  • Accessory 앱은 show()만으로 창이 앞으로 오거나 포커스되지 않음(#5122) → 노출/입력 시 set_focus() 명시.
  • set_activation_policy는 App에서 호출(과거 AppHandle 불가, #9244) — 초기화 시점 주의.
  • Spotlight/Raycast급 "활성 안 뺏는 부유"는 NSPanel nonactivatingPanel + canBecomeKeyWindow 필요 → 스톡 Tauri 미노출, objc2/커뮤니티 플러그인 필요(도입은 로드맵 결정 사항).
권고 동작

웹툰을 "보기"는 대부분 스크롤뿐이라 포커스가 늘 필요하진 않다. 따라서 평소엔 Ghost(비활성·클릭통과), 핫존에서 Revealed로 두고, 스크롤/클릭 등 입력이 실제로 필요할 때만 set_focus()로 잠깐 활성화하는 모델이 가장 단순하면서 충분하다. NSPanel 도입은 "활성 전환이 거슬린다"는 문제가 실측될 때 검토한다.
