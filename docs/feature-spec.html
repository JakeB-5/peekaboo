기능 명세 — Peekaboo

Peekaboo의 기능을 7개 영역으로 분해하고, 각 기능을 MVP / v1 / v2 우선순위로 분류한다. 항목마다 핵심 Tauri API(실제 구현에 쓰는 정확한 메서드·설정·권한)와 한계·격차를 함께 명시한다. 모든 API 이름은 사전 검증을 거친 값이며, 특히 화면 공유 비노출의 macOS 15+ 한계는 정직하게 표기한다. 동작 설계·상태 머신은 아키텍처, 정확한 코드·구조는 기술스택·구조 참조.

MVP — 최소 동작 v1 — 스텔스 완성 v2 — 편의·확장 스텔스 핵심 한계 있음

①기능 맵 개요

Peekaboo를 7개 기능 영역으로 분해한다. "스텔스 핵심"으로 표시된 영역(패닉 탈출·Hover-Reveal·앱 은폐)이 제품 가치의 중심이며, 나머지는 이를 지탱한다. MVP는 "띄우고 → 즉시 숨긴다" 직선 경로에 집중한다.

읽는 법

MVP·v1·v2 는 출시 우선순위, ★ 는 스텔스 핵심 기능이다. 핵심 Tauri API는 사전 검증된 실제 메서드/설정/권한 식별자이며, 한계는 구현 시 반드시 인지해야 하는 제약이다. 가장 중요한 한계는 ⑤의 "화면공유 비노출이 macOS 15+에서 동작하지 않음"이다.

②투명 오버레이 창 · 뷰어

웹툰 페이지를 원하는 크기·위치의 떠 있는 투명 창으로 띄우는, 모든 기능의 토대. 무테(無테)·그림자 없음·항상 위·작업표시줄 비표시로 "창처럼 보이지 않는 창"을 만든다.

투명 · 무테 · 항상-위 창

MVP
레벨 always-on-top장식 무테

제목표시줄·테두리·그림자를 모두 제거하고 배경을 투명하게 만들어, 콘텐츠만 화면에 얹힌 듯 보이게 한다. 다른 앱 위에 항상 떠 있다.

API 설정 app.windows[]: transparent:true · decorations:false · shadow:false · alwaysOnTop:true · skipTaskbar:true. 런타임 WebviewWindow::set_always_on_top(true)

한계 macOS 투명은 app.macOSPrivateApi:true + Cargo macos-private-api feature 필수 → App Store 배포 부적격(개인 배포는 무관)

크기 · 위치 조절

MVP
이동 드래그리사이즈 핸들/단축키

원하는 크기로 키우거나 줄이고, 화면 구석 등 원하는 위치로 옮긴다. 무테 창이므로 커스텀 드래그 영역을 둔다.

API set_size / set_position / start_dragging, JS getCurrentWindow().startDragging() · CSS data-tauri-drag-region

비고 핫존(호버 감지 영역)과 드래그 영역의 좌표 정의를 일관되게 관리(④와 연동)

웹툰 뷰어 로드

MVP
입력 URL저장 북마크

임의의 웹툰 URL을 창에 로드한다. 자주 보는 작품을 북마크로 저장하고 빠르게 전환한다.

API 창 설정 url(원격 URL 허용) 또는 런타임 내비게이션. 북마크는 ⑥ 영속화에 저장

한계 일부 사이트는 로그인·임베드 차단 가능 — 메인 윈도우 내비게이션으로 회피하되 사이트별 검증 필요(격차)

Spaces 전역 표시

v1
대상 모든 데스크톱

macOS Spaces(가상 데스크톱)를 전환해도 따라오게 해, 작업 공간을 바꿔도 즉시 다시 볼 수 있다.

API 설정 visibleOnAllWorkspaces:true / 런타임 set_visible_on_all_workspaces(true)

비고 풀스크린 앱 위 부유는 스톡 창으로 한계 — 필요 시 NSPanel 검토(아키텍처 참조)

③패닉 탈출 스텔스 핵심

제품의 심장. 들킬 위기에 전역 단축키 한 번으로 창이 즉시 사라진다 — 다른 앱이 포커스를 가진 상태(웹툰 창은 뒤에 있음)에서도 반드시 동작해야 한다. "기능"이 아니라 "반사신경"으로 느껴져야 한다.

전역 패닉 단축키

MVP
플러그인 global-shortcut동작 토글 hide/show

지정 단축키로 창을 즉시 숨기고, 다시 누르면 복귀한다. 포커스 여부와 무관하게 동작.

API tauri-plugin-global-shortcut — Rust Builder::new().with_handler(..).build(), app.global_shortcut().register(Shortcut::new(Some(Modifiers::SUPER | Modifiers::SHIFT), Code::KeyH)); 권한 global-shortcut:allow-register

핵심 macOS 권한 불필요 — 표준 조합은 RegisterEventHotKey 사용(시스템 중개). 단 미디어키 금지(CGEventTap → 손쉬운사용 권한 필요)

즉시성 · 흔적 최소화

MVP
지연 최소복귀 상태 보존

숨김은 체감 지연이 없어야 하고, 복귀 시 보던 위치가 유지돼야 진짜 "탈출"이다. 숨김 시 화면에 잔상이 남지 않게 한다.

설계 숨김 동작은 Rust 핸들러에서 직접 window.hide()(IPC 왕복 없이) — JS 경유보다 빠름. 복귀 위치/스크롤은 메모리 유지

주의 플러그인 이중 초기화 시 os error 22 패닉 — 등록은 메인 스레드에서 1회만

왜 이 기능이 가장 쉽고 가장 중요한가

패닉 단축키는 구현 난이도가 낮고(검증된 플러그인 + 권한 불필요), 제품 가치는 가장 크다. MVP에서 가장 먼저 동작시켜야 하는 기능이며, 화면 공유 비노출이 불완전한 macOS 15+ 환경에서는 사실상 스텔스의 최후 방어선 역할까지 겸한다(화면 공유 전에 미리 숨기기).

④투명도 · Hover-Reveal 스텔스 핵심

"평소엔 거의 안 보이고, 보려는 영역에 마우스를 올릴 때만 또렷해진다." 투명도로 평소 노출을 낮추고, hover-reveal로 필요할 때만 콘텐츠를 드러낸다. 단, 클릭 통과와 결합되면 hover 감지 방식이 통상적 방법과 달라진다.

투명도 조절

MVP
방식 CSS alpha범위 0–100%

평소 불투명도를 슬라이더로 조절한다. 낮출수록 덜 보이고, 높일수록 잘 보인다. 평소값과 호버시값을 따로 둔다.

API 투명 창(transparent:true) 위에서 콘텐츠 컨테이너의 CSS opacity/배경 rgba() alpha 조절

한계 창 전체 alpha를 직접 바꾸는 안정적 공개 API는 미확정 — CSS 레벨에서 처리(확인필요: 창 alpha 네이티브 API 필요 시 별도 조사)

Hover-Reveal (클릭 통과)

v1
감지 커서 폴링통과 ignore cursor

평소엔 창이 마우스 클릭을 아래 앱으로 통과시켜(있어도 방해 안 됨), 핫존에 커서가 들어오면 또렷해지고 상호작용을 받는다.

API 클릭통과 set_ignore_cursor_events(bool) / JS setIgnoreCursorEvents. 감지 Window::cursor_position()→PhysicalPosition<f64> / JS cursorPosition(). 권한 core:window:allow-set-ignore-cursor-events · core:window:allow-cursor-position

핵심 한계 클릭통과 중에는 DOM mouseenter/mouseleave는 물론 네이티브 CursorEntered/Left 이벤트도 발생하지 않음 → 커서 위치 폴링(~30–60ms) + 핫존 rect 히트테스트가 유일한 정공법

설계 크럭스 · 클릭통과 ↔ 호버의 모순

"클릭은 통과시키되 호버는 감지"는 서로 충돌한다. 클릭통과를 켜는 순간 창은 마우스 이벤트를 전혀 받지 못하기 때문이다. 검증된 해법은 전역 커서 좌표를 주기적으로 읽어(cursor_position()) 핫존 사각형과 비교하는 폴링 루프다. 이 좌표 API는 "어느 창이 커서를 소유하는가"와 무관한 데스크톱 절대좌표라 클릭통과 상태에서도 정상 동작한다. 폴링 비용을 줄이려면 핫존만 클릭통과를 해제하거나 OS 레벨 NSEvent 모니터(objc2)로 이벤트 기반 전환도 가능하나, 순수 Tauri 범위에서는 폴링이 표준이다. 루프 설계는 아키텍처 참조.

⑤화면공유 비노출 · 앱 은폐

앱의 "존재" 자체를 감추는 두 기능. 앱 은폐(Dock·Cmd-Tab)는 확실하게 동작하지만, 화면 공유 비노출은 macOS 버전에 따라 한계가 분명하다 — 이 차이를 정직하게 구분해 둔다.

현실 점검 · 화면 공유 비노출은 macOS 15+에서 동작하지 않는다

사전 검증(Tauri #14200) 결과, setContentProtected(true)는 NSWindow.sharingType=NSWindowSharingNone으로 매핑되는데, macOS 15(Sequoia)+의 ScreenCaptureKit는 합성 프레임버퍼를 캡처하며 이 플래그를 무시한다. Zoom·Meet·QuickTime는 모두 ScreenCaptureKit를 쓰므로 오버레이가 화면 공유에 그대로 노출된다. content protection은 macOS ≤14 + 구형 CoreGraphics 캡처에서만 유효하며(심지어 일부 Chromium 기반 공유는 구버전에서도 못 막음 — #10213), Apple의 의도된 변경이라 알려진 우회가 없다. 따라서 이 기능은 best-effort 보조 수단으로만 포지셔닝하고, 화면 공유 환경의 실질 방어는 ③ 패닉 단축키(사전/즉시 숨김)에 둔다.

화면공유 비노출(content protection)

v1
매핑 sharingType=none유효 macOS ≤14

창을 화면 캡처 대상에서 제외하도록 요청한다. 구버전 macOS에서는 화면 공유·스크린샷에서 빠지지만, 최신 macOS에서는 효과가 없다.

API Rust WebviewWindow::set_content_protected(true) / JS getCurrentWindow().setContentProtected(true) / 설정 contentProtected:true. 권한 core:window:allow-set-content-protected

한계 macOS 15+에서 무효(ScreenCaptureKit가 무시). 보조 수단으로만 사용하고 사용자에게 한계를 안내

앱 은폐 (Dock · Cmd-Tab)

v1 스텔스 핵심
정책 Accessory대상 Dock·앱스위처·메뉴바

앱을 Dock 아이콘·Cmd-Tab 목록·메뉴바에서 모두 제거해, 앱이 실행 중이라는 단서 자체를 없앤다. 이건 macOS에서 확실히 동작한다.

API 설정 app.macOS.activationPolicy:"accessory"(정확 경로) 또는 런타임 app.set_activation_policy(ActivationPolicy::Accessory). 정적 대안 Info.plist LSUIElement=true

주의 Accessory 앱은 show()만으로 포커스가 안 옴 → 입력 필요 시 set_focus()로 온디맨드 활성화(포커스 모델 참조)

⑥설정 · 상태 영속화

크기·투명도·핫존·단축키 등 사용자 취향을 저장하고, 재실행 시 그대로 복원한다. 설정 UI는 호출했을 때만 뜨는 가벼운 패널로 둔다.

설정 항목

  • 창 — 기본 크기·위치, 시작 시 표시 여부
  • 투명도 — 평소 불투명도 / 호버 시 불투명도
  • 핫존 — 호버 감지 영역(위치·크기), 다중 가능(v2)
  • 단축키 — 패닉 토글 키, 표시/숨김 키 매핑
  • 뷰어 — 북마크(작품 URL) 목록

영속화

설정·북마크·마지막 보던 위치를 디스크에 저장하고 재실행 시 복원한다.

  • 저장 — tauri-plugin-store(JSON KV) 또는 fs 플러그인
  • 대상 — 설정값 · 북마크 · 마지막 URL/스크롤
  • 주의 — 파일 경로/이름이 "흔적"이 되지 않도록 평범하게(스텔스 관점)

⑦확장 (v2)

핵심 스텔스가 완성된 뒤 추가하는 편의·고급 기능. 없어도 제품은 성립하므로 후순위.

우선순위 요약 (전 영역)

MVP는 "투명 오버레이 + 뷰어 로드 + 패닉 단축키 + 투명도"로 "띄우고 즉시 숨긴다"를 완성한다. v1에서 Hover-Reveal·앱 은폐·content protection(보조)·설정 영속화로 스텔스를 완성하고, v2는 편의 기능이다.
